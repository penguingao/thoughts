If we are convinced that C++ coroutine makes it easier to write and reasonable
correct Envoy HTTP filters, the next step is to find a good set of primitives
that we can safely use in Envoy's code base. This is because C++20 only provides
language level keywords to make it possible to implement coroutine, but it
doesn't come with standard library.

C++ doesn't have a default executor or thread pool. The async nature of
coroutine requires a way to specifiy "how and where to resume a suspended
coroutine". This means, we need to glue Envoy's `Dispatcher` into a library.

Before we decide whether to create our own library, or using a third party
library like [Asio](https://think-async.com/Asio/), we should first lay out what
we want in Envoy from ease of use, and safety point of view.

# What semantics do we need?

Given that Envoy has its own `Dispatcher` as the main runtime environment of
the majority of the code, we prefer to run coroutines on the same `Dispatcher`
of the caller (a coroutine, or normal function) as much as possible so that it's
compatible with the rest of the code base.

When coroutine exists in the code base, there is always going to be an entry
point where the caller is not a coroutine, because `main()` cannot be a
coroutine. On the other hand, we almost always have to end a chain of coroutine
call with a non-coroutine awaitable when we need to `co_await` on an event
that's managed by either OS (`epoll`, `event fd`, `io_uring`) or another async
/ callback library (`c-ares`, `openssl`).

## Ownership model
To avoid life cycle problems, we want a coroutine to have clear and consistent
memory ownership model throughout the chain of coroutines.

`co_await` gives us a way for the caller to always outlive the coroutines or
awaitables it starts. This makes the memory model clean, because we an async
event eventually arise, it always have a place to resume execution so that
whatever resources associated with this event can be cleaned up.

This however, raises the challenge of potential indefinite `co_await`. We will
address this in later sections.

## Calling into callback code
A chain of coroutine always end with waiting on some non-coroutine events, wnen
this happens, the library should:
- provide a way callback driven awaitable that can be `co_awaited` on.
- provide a timeout on the awaitable in case the callback event never
  arises.
- cancel the pending event and free up the resoures once the timeout expires.

Callback events typically have cancellation contracts, so this means we provide
some good templates / base classes to implement such cancellation contracts.

Having the timeout is important because we don't want the chain of `co_await` to
hang indefinitely.

With this, we by and large avoided the risk of indefinite await problem.

## Calling from callback code
A chain of coroutine has to be lanched from a non-coroutine caller eventually.
From the non-coroutine caller's point view, the library should have:
- a way to start a coroutine asynchronously without a `co_await`.
  this is also useful for coroutine caller to use run multiple coroutines - more
  discussions in the "Strucuted concurrentcy" section.
- a way to receive a callback of a running coroutine when it's done.
- when the caller reaches end of life, the asynchronously running coroutine
  should not crash the program or cause a memory leak.

If we assume a chain of coroutines will always finish, the non-coroutine caller
doesn't need to actively cancel the coroutines it launches. It needs to be able
to hand the ownership to some other objects that can babysitter the coroutines
until they finish.

In the mean time, Implementing cancellation is challenging any way because it's
implicity control point injection into the coroutine body (if a coroutine is
cancelled at a suspension point, and it has logic to clean up resources after
that suspension point, it will not have a chance to do it).

The only exception is the leaf awaitables that interfaces with the callback
events, which does usually have reasonable cancellation semantics so we can
add timeout driven cancellation.

## Structured concurrency
It is desirable to have strucuted concurrency so that a coroutine can launch
parallel tasks (coroutines or awaitables) and await on all of any of them. This
is useful to implement request hedging or many types of timeouts. An coroutine /
awaitable with a timeout can also be modelled as `co_await any_of(my_coroutine,
timeout)`.

This requires the library to support an awaitable `any_of`. The need of `any_of`
mandates the proper handling of unfinished coroutines. We've established that
cancellation is hard on coroutines. This means we should again rely on ownership
transfer.

We also need `all_of`, but that doesn't come with ownership and life cycle
problems.

## Cross thread tasks
This is probably not the first class concern, but even in the siloed environment
like Envoy, we still have the need for this. Currently, this is facilitated
mainly by `Dispatcher::post`.

This is mainly used between main thread and worker thread to exchange control
plane events (xDS push) / administrative events (overload manager, stats) / DNS.

We would like the coroutines to by default inherit its caller's `Dispatcher` as
it's the majority of the case. We also want to be able to specify a `Dispatcher`
that the coroutine should run on, but this should come as an override.

## Froeign executor / runtime
Another typical pattern is dealing with foreign runtime - a non-`Dispatcher`
thread.

If we can generalize `Dispatcher` in to an `Coroutine::Executor` interface, we
can then wrap foreign runtime into this `Executor` to provide the same cross
thread task support that is covered in the earlier section.

## Debuggability
On a crash in coroutine, we would like to be able to trace both the coroutine
call stack by walking the coroutine handle pointers, and the resumption call
stack. This helps us btter triangulate problems as we can see states of the
callers as well as the events.

In the callback world, we only see "resumption", and it's typically hard to find
the caller's stack frame to exam its state.
