If we are convinced that C++ coroutines make it easier to write and reason about
Envoy HTTP filter code, the next step is to find a good set of primitives
that we can safely use in Envoy's codebase. This is because C++20 only provides
language-level keywords to enable implementing coroutines, but does not include
a standard coroutine library (e.g., standard executors and tasks).

C++ doesn't have a default executor or thread pool. The async nature of
coroutines requires a way to specify "how and where to resume a suspended
coroutine". This means we need to integrate Envoy's `Dispatcher` into a library.

Before deciding whether to create our own library or use a third-party
library like [Asio](https://think-async.com/Asio/), we should first lay out
our requirements for Envoy from both an ease-of-use and safety point of view.

# What semantics do we need?

Given that Envoy has its own `Dispatcher` as the main runtime environment for
the majority of the code, we prefer to run coroutines on the same `Dispatcher`
as the caller (whether a coroutine or normal function) as much as possible so
that it is compatible with the rest of the codebase.

When coroutines exist in the codebase, there will always be an entry
point where the caller is not a coroutine, because `main()` (or event callbacks)
cannot be a coroutine. On the other hand, we almost always have to end a chain
of coroutine calls with a non-coroutine awaitable when we need to `co_await` on
an event that is managed by either the OS (`epoll`, `eventfd`, `io_uring`) or
another async/callback library (`c-ares`, `openssl`).

## Ownership model
To avoid lifecycle problems, we want coroutines to have a clear and consistent
memory ownership model throughout the chain of coroutines.

`co_await` guarantees that the caller outlives the coroutines or awaitables it
awaits. This keeps the memory model clean because when an async event eventually
arises, it always has a valid frame to resume execution so that whatever resources
associated with this event can be cleaned up.

This, however, raises the challenge of potential indefinite `co_await`s. We will
address this in later sections.

## Calling into callback code
A chain of coroutines always ends with waiting on non-coroutine events. When
this happens, the library should:
- Provide a callback-driven awaitable that can be `co_awaited`.
- Provide a timeout on the awaitable in case the callback event never arises.
- Cancel the pending event and free up resources once the timeout expires.

Callback events typically have cancellation contracts, which means we should
provide good templates/base classes to implement such cancellation contracts.

Having a timeout is essential because we don't want the chain of `co_await` calls
to hang indefinitely.

With this, we largely avoid the risk of the indefinite await problem.

## Calling from callback code
A chain of coroutines must eventually be launched from a non-coroutine caller.
From the non-coroutine caller's point of view, the library should provide:
- A way to start a coroutine asynchronously without a `co_await`.
  (This is also useful for a coroutine caller to run multiple coroutines—see
  further discussion in the "Structured concurrency" section).
- A way to receive a callback from a running coroutine when it finishes.
- Safety guarantees so that when the caller reaches its end of life, the asynchronously
  running coroutine does not crash the program or cause a memory leak.

If we assume a chain of coroutines will always finish, the non-coroutine caller
doesn't need to actively cancel the coroutines it launches. It needs to be able
to hand ownership to another object that can babysit the coroutines until they
finish.

In the meantime, implementing cancellation is challenging because it involves
implicit control-flow injection into the coroutine body (if a coroutine is cancelled
at a suspension point and has cleanup logic following it, it will not get a chance
to execute that logic).

The only exception lies in leaf awaitables that interface with callback events,
which usually have reasonable cancellation semantics suited for timeout-driven
cancellation.

## Structured concurrency
It is desirable to have structured concurrency so that a coroutine can launch
parallel tasks (coroutines or awaitables) and await all or any of them. This is
useful to implement request hedging or various timeout patterns. A coroutine/awaitable
with a timeout can also be modeled as `co_await any_of(my_coroutine, timeout)`.

This requires the library to support an awaitable `any_of`. The need for `any_of`
mandates proper handling of unfinished coroutines. Since cancellation is hard on
coroutines, we should again rely on ownership transfer.

We also need `all_of`, which does not carry the same ownership and lifecycle complexities.

## Cross-thread tasks
While cross-thread task execution is not the primary concern in Envoy's thread-siloed
architecture, it remains a necessity. Currently, this is facilitated mainly by
`Dispatcher::post`.

This is primarily used between the main thread and worker threads to exchange
control plane events (xDS push), administrative signals (overload manager, stats),
or DNS resolution.

By default, coroutines should inherit their caller's `Dispatcher`, as that covers
the majority of cases. We also want to allow explicitly specifying a `Dispatcher`
for the coroutine to run on as an override.

## Foreign executor / runtime
Another common pattern involves dealing with foreign runtimes—threads not managed
by an Envoy `Dispatcher`.

If we can generalize `Dispatcher` into a `Coroutine::Executor` interface, we can
then wrap foreign runtimes in this `Executor` interface to provide the same cross-thread
task support described in the previous section.

## Debuggability
Upon a crash within a coroutine, we want to be able to trace both the coroutine call
stack (by walking coroutine handle pointers) and the physical resumption call stack.
This helps us better triangulate issues by exposing caller state alongside event state.

In the callback world, we only see the resumption stack frame, making it difficult
to inspect the caller's state.
